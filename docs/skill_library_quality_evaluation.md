# 技能库质量与重复/混乱评估方案

## 一、背景与问题

当前逻辑：**只要 Critic 判定任务成功，就会把本次执行的代码抽象成技能并写入或覆盖（upsert by name）**。可能带来的情况：

- **同名覆盖**：同一抽象名（如 `chop_wood`）被多次成功任务覆盖，最终只保留最后一次的 code/description，可能和早期版本语义不一致。
- **异名同效**：不同任务抽象出不同 name（如 `mine_oak_log` 与 `chop_wood`），但代码逻辑高度相似甚至等价，检索时会出现多条语义重叠的技能，增加上下文噪音、稀释高质量技能。
- **效果与 Critic 不一致**：Critic 判成功但实际游戏效果未达预期（如只挖了 1 块却判通过），错误技能被写入，若不剔除会持续被检索到并误导后续生成。

本方案仅做**评估与改进方向**，不要求立刻改代码；可与「导出 + 人工删除」配合使用，逐步保证技能库正确性。

---

## 二、评估维度

### 2.1 重复与语义重叠

- **指标**  
  - 技能两两之间的 **description/code 语义相似度**（如用现有 embedding 算余弦相似度）。  
  - 若相似度高于某阈值（如 0.85），可视为「潜在重复或强重叠」。
- **做法**  
  - 周期性（如每周）对 `list_all_skills()` 或导出列表做一次两两相似度计算（或只对 description + code 前 500 字做 embedding 再比较）。  
  - 输出「相似对」列表，人工复核：保留一条、合并描述或删除冗余。
- **可选自动化**  
  - 在写入前做「与已有技能相似度检查」：若新技能与某条已有技能相似度 > 阈值，则提示「可能重复」或走「更新已有」而非新增一条异名技能（需产品化策略，此处仅作思路）。

### 2.2 同名覆盖导致的历史漂移

- **指标**  
  - 同一 `name` 被覆盖的次数无法从当前 ChromaDB 直接看出（无版本历史）。  
  - 可观察：**同一 name 下当前 code/description 与「最近一次成功任务」的原始 task 是否一致**（若保留任务日志，可对比）。
- **做法**  
  - 若将来为技能库加「版本」或「最近一次来源 task」的元数据，可评估：同名多次覆盖后，当前技能是否仍符合你对「该 name」的预期。  
  - 短期：通过**导出技能库**（`export_skills.py`）做人工抽查，对高频使用的 name 看 code 是否与预期一致。

### 2.3 Critic 通过但实际效果不佳

- **指标**  
  - 人工复核：在游戏中**直接复用某条技能对应的行为**（或让 Agent 执行与技能描述一致的任务），观察是否达到预期。  
  - 若「导出 → 按 name 删除」后，同类任务表现变好，说明该技能为负向样本。
- **做法**  
  - 使用 **`remove_skill.py`** 按 name 删除已确认的不良技能。  
  - 可选：在技能元数据中增加「人工标记：保留/待删/已删」字段，便于批量评估（需改存储 schema，此处仅作方向）。

### 2.4 检索质量与上下文噪音

- **指标**  
  - 对固定一组**测试 query**（如「挖 5 个橡木」「合成工作台」），记录每次检索到的 top_k 技能 name 与 score。  
  - 若同一 query 多次检索结果中，**低质量或无关技能**经常混入 top_k，说明库内噪音大或重复多。
- **做法**  
  - 定期用 `get_programs_string(query, top_k=5)` 对 5–10 个典型 query 跑一遍，人工给「是否相关、是否愿意被生成参考」打分，统计「相关技能占比」或「无关/重复条数」。

---

## 三、建议的评估节奏（不实现也可按此思路执行）

1. **每周或每积累 N 条新技能后**  
   - 运行 `export_skills.py`，浏览 `data/skill_db_export/` 中的 .js 与 .json。  
   - 对「效果存疑」的技能在游戏中做 1–2 次复用或等价任务验证，不合格则 `remove_skill.py <name>` 删除。
2. **每月或技能数 > 某规模（如 30）**  
   - 做一次「两两相似度」扫描（用现有 embedding 即可），列出高相似对，人工决定合并或删冗余。  
   - 对 5–10 个固定 query 跑检索，记录 top_k 结果并打分，观察是否随技能增多而质量下降。
3. **长期**  
   - 若发现同名覆盖导致语义漂移明显，再考虑为技能增加「版本」或「来源 task」等元数据，或「写入前与已有高相似则更新而非新增」的策略。

---

## 四、小结

- **重复与混乱**：主要来自「异名同效」和「同名多次覆盖」，可通过**相似度评估 + 人工合并/删除**和**导出后抽查**来缓解。  
- **绝对正确性**：依赖你在实际游戏中**复用或等价验证**后，用 **`remove_skill.py`** 剔除不良技能；导出脚本便于你看到每条技能的代码与描述，便于做上述判断。  
- 本方案为评估与流程建议，具体阈值、频率和是否做自动化合并/去重，可按需要再细化或落地到代码。
